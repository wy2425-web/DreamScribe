Design Evolution

v1 — Text-First

Initial concept was a text input box (like a journal app). User testing revealed people don't want to type right after waking — it's too slow and the dream fades.

v2 — Voice-First (Current)

Moved to a large mic button as the primary input, with text as a fallback. Two short follow-up questions reduce blank-page anxiety and help the AI extract mood and key moments.

Key UX Principles

1

Speed over perfection

The user should be able to describe their dream in under 60 seconds. The AI fills in the rest.

2

Low cognitive load on waking

Only 2 follow-up questions. No dropdowns, no settings, no menus before the comic is generated.

3

Delight through the unexpected

The AI auto-selects an art style matching the dream's mood — users are surprised to see their dream rendered in a style that "just fits."

AI / ML Models

Model Type Example Purpose
Large Language Model GPT-4 / Claude Parses dream text, fills narrative gaps, writes dialogue in the user's voice, structures the 6-beat story arc.
Text-to-Image DALL-E 3 / Stable Diffusion Generates each comic panel from scene descriptions with style conditioning for visual consistency.
Sentiment / Mood Classifier Fine-tuned BERT Classifies dream tone (playful, eerie, surreal, heartwarming) to drive art style and color palette.
Speech-to-Text (ASR) OpenAI Whisper Transcribes voice descriptions into clean text — tuned to handle groggy, half-awake speech patterns.
Style Consistency LoRA / IP-Adapter Keeps character faces, settings, and visual style coherent across all 6 separately-generated panels.

Prompting Strategy

Dream Parsing Prompt

Instructs GPT-4 to extract characters, setting, key events, mood, and art style as structured JSON. Uses few-shot examples of surreal dream narratives to ground the model.

Story Engine Prompt

Uses a 6-beat story arc template (Setup → Rising Action → Complication → Climax → Resolution → Coda). The user's own phrasing is injected into the system prompt so dialogue matches their voice.

def dream_to_comic(dream_input: DreamInput, output_dir: str = "dreamscribe_output") -> DreamComic:
    """Full DreamScribe pipeline: DreamInput → ParsedDream → Story Beats → Comic"""
    os.makedirs(output_dir, exist_ok=True)

    # Step 2: Parse the dream with GPT-4
    parsed = parse_dream(dream_input)

    # Step 3: Build 6-beat story arc
    beats = build_story_arc(parsed, dream_input)

    # Step 4: Generate comic panels (DALL-E 3)
    panels = []
    for i, beat in enumerate(beats, 1):
        image_prompt = build_image_prompt(beat["scene_description"], parsed.art_style)
        panel = ComicPanel(
            panel_number=i,
            scene_description=beat["scene_description"],
            dialogue=beat["dialogue"],
            image_prompt=image_prompt
        )
        panel.image_path = generate_panel_image(panel, parsed.art_style, output_dir)
        panels.append(panel)

    # Assemble and save the comic
    comic = DreamComic(title="Dream of the Cloud City", panels=panels,
                       mood=parsed.mood, art_style=parsed.art_style)
    return comic

parse_dream() — real GPT-4 implementation 
# Real implementation (swap mock for this when OPENAI_API_KEY is set)
import openai
client = openai.OpenAI(api_key=os.environ["OPENAI_API_KEY"])

prompt = f"""
Parse this dream into structured components. Return JSON only.
Dream: {dream_input.raw_text}
Weirdest part: {dream_input.weirdest_part}
Emotional tone: {dream_input.emotional_tone}

Return:
{{
  "characters": [...],
  "setting": "...",
  "key_events": [...],
  "mood": "playful|surreal|dark|heartwarming",
  "art_style": "..."
}}
"""

response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": prompt}],
    response_format={"type": "json_object"}
)
data = json.loads(response.choices[0].message.content)
return ParsedDream(**data)
05 — Troubleshooting Log

Challenges & How I Fixed Them

✓ Resolved
GPT-4 invented major plot elements not in the dream
Fix: Added explicit prompt constraints — "only use details from the user's description; mark any gap-filling with [INFERRED]." This dramatically reduced hallucinations while still allowing creative gap-filling where needed.
✓ Resolved
DALL-E 3 added text and speech bubbles into the images
Fix: Added the explicit instruction "No text, no speech bubbles, no captions" to every image prompt prefix. Consistency improved significantly.
✓ Resolved
Mood classifier was over-classifying everything as "surreal"
Fix: The keyword-based classifier was too broad. Switched to a weighted scoring approach that requires at least 2 distinct signals before assigning a mood. Also added "heartwarming" as an explicit default for dreams with no strong negative valence.
⏳ In Progress
Visual consistency across panels — characters look different in each panel
Current approach: Adding a consistent character description seed phrase to every panel prompt (e.g. "same character: short dark hair, purple jacket, 20s"). Exploring LoRA fine-tuning and IP-Adapter as longer-term solutions.
⏳ In Progress
Whisper struggles with groggy, mumbled speech right after waking
Current approach: Added a post-transcription GPT-4 cleanup pass that fixes filler words, run-on sentences, and incomplete thoughts common in half-awake speech. Testing with deliberately low-quality audio samples.
06 — Instructor Support

Open Questions for the Instructor

These are the specific areas where I'd love guidance before the final submission:

Question 01
How do I maintain visual consistency across all 6 DALL-E panels?
LoRA / IP-Adapter seems promising but I haven't set up the workflow yet. Is there a simpler seed-based approach that works well with DALL-E 3, or should I switch to Stable Diffusion for this use case?
Question 02
How do I prompt an LLM to build a coherent story from fragmented, nonlinear dream input without over-inventing?
Dreams are inherently nonlinear and vague. I'm looking for the right balance between filling narrative gaps creatively and staying true to what the user actually described.
Question 03
What's the right UX pattern for real-time story editing mid-generation?
I want users to be able to redirect the story while it's generating. How should I handle partial regeneration of only the affected panels without regenerating all 6?
Question 04 — (from Prof. Safinah)
What ML approach makes sense for longitudinal dream analysis?
Is topic modeling or embedding similarity the right approach for helping users spot patterns in their dreams over time? How do I handle privacy — should this analysis be local-only?
Question 05 — (from Prof. Maria)
How do I handle nightmares and dark dream content safely?
Some users will have dark or distressing dreams. How do I balance depicting those faithfully with ensuring the content moderation layer doesn't strip out the emotional truth of the dream?
